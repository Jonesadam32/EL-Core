# EL Core v1.22.0 — Testing Guide
## Definition Consensus Review System

**Upload:** WordPress Admin → Plugins → Add New → Upload Plugin → `releases/el-core.zip`

---

## Pre-Test Setup

You need at least one project with:
- A project definition saved (Discovery tab has content in the fields)
- At least 2 stakeholders: one Decision Maker, one Contributor

If you don't have this, create a project, add definition content, add stakeholders before starting.

---

## Test 1 — Admin: Discovery Tab Status Bar

1. Go to a project → **Discovery tab**
2. **Verify:** Status bar appears at the top showing:
   - "Definition Status:" label + **Draft** badge (grey)
   - "Send to Client for Review" button (primary blue)
   - "Confirm & Lock Definition" button (secondary)
3. **Verify:** The old lock button at the bottom of the form is gone (it's now only at the top)

✅ Pass if: Status bar renders, both buttons visible, no duplicate lock button at bottom

---

## Test 2 — Admin: Send for Review Modal

1. Click **"Send to Client for Review"**
2. **Verify:** Modal opens with:
   - Info notice explaining the workflow
   - Deadline date field pre-filled to 7 days from today
   - "Send for Review" submit button
3. Change the deadline to a date 3 days from now
4. Click **"Send for Review"**
5. **Verify:** Success alert appears → page reloads
6. **Verify:** Status bar now shows **"Sent for Review"** badge (blue) + the deadline date

✅ Pass if: Modal opens, form submits, page reloads with correct status badge

---

## Test 3 — Admin: Status Badge After Sending

1. Still on the Discovery tab after Test 2
2. **Verify:** Status badge shows "Sent for Review" (blue/info)
3. **Verify:** "Send to Client for Review" button is still present (can re-send)
4. **Verify:** "Confirm & Lock Definition" button is still present (admin can override-lock anytime)
5. **Verify:** Deadline date is shown next to the badge (e.g., "Deadline: Mar 3, 2026")

✅ Pass if: All three elements visible, deadline date correct

---

## Test 4 — Client Portal: Definition Card with Review Open

1. Log in as a **Contributor** stakeholder (use Login As from the Stakeholders tab)
2. Navigate to the portal page
3. **Verify:** The Definition card appears in Stage 1 (even though definition is NOT locked)
4. **Verify:** The card shows a "Review Open" badge in blue
5. **Verify:** Card subtitle says "Review what we're building"

✅ Pass if: Card appears, shows "Review Open" badge

---

## Test 5 — Portal: Definition Review Modal (Contributor)

1. Still logged in as Contributor
2. Click the **Definition card** to open the modal
3. **Verify:** Modal opens (large/scrollable)
4. **Verify:** "Review Open" badge in header
5. **Verify:** Countdown timer shows (e.g., "2d 23h remaining")
6. **Verify:** Intro text says "Review each section below..."
7. **Verify:** All definition fields are visible (Site Description, Primary Goal, etc.)
8. **Verify:** Each field has:
   - A field label in small caps
   - The field value text
   - "+ Add comment" toggle button
   - "Looks good" and "Needs revision" verdict buttons
9. **Verify:** "Submit My Input" button is **disabled** with hint text "Scroll through all fields..."

✅ Pass if: Modal renders with all fields and verdict buttons; Submit button disabled

---

## Test 6 — Portal: Scroll-Depth Gate

1. In the definition review modal (Contributor, review open)
2. **Without scrolling** — verify Submit button is still disabled
3. Scroll all the way to the bottom of the modal body
4. **Verify:** "Submit My Input" button becomes **enabled** (no longer greyed out)
5. The hint text should disappear

✅ Pass if: Button enables only after scrolling to the bottom

---

## Test 7 — Portal: Add Comment on a Field

1. In the modal, click **"+ Add comment"** on the first field (Site Description)
2. **Verify:** Textarea expands inline below the toggle
3. Type: "This description looks accurate to me."
4. Click **"Post Comment"**
5. **Verify:** Comment appears in the thread with your name and timestamp
6. **Verify:** The "+ Add comment" toggle reappears
7. Add a second comment on a different field

✅ Pass if: Comment posts and renders without page reload

---

## Test 8 — Portal: Threaded Reply

1. In the modal, find a comment posted in Test 7
2. Click **"Reply"** beneath that comment
3. **Verify:** Reply textarea appears
4. Type: "Agreed, that covers it well."
5. Click **"Post Reply"**
6. **Verify:** Reply appears nested/indented under the original comment
7. Click **"Reply"** again and click **"Cancel"** — verify form closes

✅ Pass if: Reply posts nested under parent comment; Cancel works

---

## Test 9 — Portal: Verdict Buttons (Contributor)

1. In the modal, find the "Site Description" field
2. Click **"Looks good"**
3. **Verify:** Button turns green/active
4. **Verify:** A "Looks good" badge appears in the field header
5. Find a different field, click **"Needs revision"**
6. **Verify:** Button turns amber/active
7. **Verify:** "Needs revision" badge appears in that field's header
8. Click **"Looks good"** on the same "Needs revision" field — verify it switches to green

✅ Pass if: Buttons toggle correctly, badges update in field header, API calls succeed (no error alerts)

---

## Test 10 — Admin: Sees Contributor Comments and Verdicts

1. Switch back to admin (use WP toolbar or switch-back)
2. Go to the project → Discovery tab
3. **Verify:** DM verdict summary card appears showing count of approvals and revision requests
4. **Verify:** Fields that received comments show a comment thread panel below the field input
5. **Verify:** Comment text, author name, and date visible
6. **Verify:** Fields with verdicts show verdict badges (green ✓ / amber "needs revision")

✅ Pass if: Summary card visible; comments appear per-field; verdicts shown

---

## Test 11 — Portal: DM Final Decision

1. Log in as the **Decision Maker** stakeholder
2. Open the portal → click Definition card
3. **Verify:** All fields show (same as Contributor view)
4. **Verify:** DM can also set per-field verdicts and post comments
5. Scroll to the bottom
6. **Verify:** "Make Final Decision" section appears with gradient background — Contributors do NOT see this
7. **Verify:** Two buttons: "Accept — Definition Approved" (primary) and "Request Revisions" (secondary)
8. **Verify:** Note textarea is present
9. Type a note: "Looks good overall, proceeding."
10. Click **"Accept — Definition Approved"**
11. **Verify:** Confirmation dialog appears
12. Confirm → **Verify:** Success alert → page reloads

✅ Pass if: DM section visible only to DM; Accept button works and reloads page

---

## Test 12 — Post-Decision States

1. After DM accepted (Test 11), log in as the **Contributor** again
2. Open the portal → click the Definition card
3. **Verify:** Modal shows "Approved" badge in header (not "Review Open")
4. **Verify:** Green "Definition Approved" banner at top of modal with the DM's note
5. **Verify:** No verdict buttons visible (review closed)
6. **Verify:** No "+ Add comment" toggles (review closed)

✅ Pass if: Approved banner shows with DM note; verdict buttons hidden

---

## Test 13 — Admin: Status After DM Approval

1. Switch back to admin
2. Go to Discovery tab
3. **Verify:** Status badge now shows **"Client Approved"** (green)
4. **Verify:** "Confirm & Lock Definition" button still present
5. Click "Confirm & Lock Definition" → confirm the prompt
6. **Verify:** Page reloads showing **"Locked"** notice at top
7. **Verify:** Status badge shows **"Locked"** (green)
8. **Verify:** Definition form fields are now read-only (no editing)

✅ Pass if: Approved → Locked flow works; fields become read-only after lock

---

## Test 14 — Needs Revision Flow (optional)

To test the revision path:
1. Create/reset a project with definition in "Sent for Review" state
2. As DM: click **"Request Revisions"** with a note: "Please expand the primary goal."
3. **Verify:** Page reloads
4. As admin: status badge shows **"Needs Revision"** (amber/warning)
5. As Contributor: open Definition card → **Verify:** "Revisions Requested" amber banner with DM note
6. As admin: edit definition fields → save → send for review again (new round)
7. **Verify:** New review opens, countdown resets, previous comments still visible

✅ Pass if: Revision banner shows with DM note; admin can re-send for review

---

## Test 15 — No Regression: Existing Features

1. Confirm the mood board voting still works (open a project with an active review session)
2. Confirm proposals still display in the portal
3. Confirm Stage navigation still works in the portal
4. Confirm Advance Stage still works from admin

✅ Pass if: No regressions in existing functionality

---

## Quick Smoke Test (if short on time)

Run Tests 1, 2, 4, 5, 7, 9, 11, 13 only — covers the full happy path.

---

## Known Limitations (not bugs)

- "Updated" badge on fields changed since last round — not built yet (planned for future)
- Reminder emails at 50% deadline and 24h before — not built yet (planned)
- Comments from a closed round are not shown in the next round's portal view (fresh start per round)
